# NiceTransfer — Development Overview

Source and issues: [github.com/joko-zauberzeug/nicetransfer](https://github.com/joko-zauberzeug/nicetransfer)

---

## Philosophy

NiceTransfer is deliberately minimal. The goal is a tool you can clone, run, and understand — no build pipeline, no package structure, no framework beyond NiceGUI. Everything needed to run lives in the project directory.

---

## Project files

| File | Purpose |
|------|---------|
| `nicetransfer.py` | The entire application — UI, server, MCP, REST endpoints |
| `nicetransfer.css` | Component styles loaded once at startup, served as inline `<style>` |
| `MANUAL.md` | User-facing documentation, rendered at `/manual` |
| `CHANGELOG.md` | Version history, rendered at `/changelog` |
| `DEVELOPMENT.md` | This file, rendered at `/development` |
| `config.toml` | Runtime configuration (generated by `install.sh`) |
| `install.sh` | Sets up the venv, generates `config.toml` and `run.sh` |
| `run.sh` | Activates the venv and starts `nicetransfer.py` (generated by `install.sh`) |

The Markdown and CSS files are read at request time, not baked in — edit them while the server is running and the next page load reflects the change.

## Design concepts

### Token model

Every request — browser, download, preview, MCP, AI discovery — requires `?token=TOKEN`. The token is printed in the startup banner and regenerates on each start unless a fixed value is set in `config.toml`. There are no public endpoints except NiceGUI internals. This is why every generated link in the code includes the token.

### Server vs. client

NiceTransfer distinguishes between the *server* (the device running the app, accessing via `127.0.0.1`) and *clients* (other devices on the same network). The server gets the full control panel; clients see only the file sections. Client permissions (delete, trash access) are configurable.

### Sections

Four file sections — Share, Upload only, Download only, Trash — each backed by a configurable directory. Deleted files always go to Trash first; nothing is removed immediately. Sections can be toggled at runtime via the control panel.

## AI integration

NiceTransfer embeds AI discovery hints in every page `<head>`:

```html
<meta name="mcp-server"      content="http://<ip>:<port>/mcp?token=...">
<meta name="mcp-server-card" content="http://<ip>:<port>/.well-known/mcp/server-card.json?token=...">
<meta name="llms-txt"        content="http://<ip>:<port>/llms.txt?token=...">
```

These three endpoints are all dynamic — served by the running instance, token-protected, and generated from the current state. An AI assistant that scans the QR code or opens the URL discovers them automatically.

A second discovery path targets terminal-based AI agents (like Claude Code): the startup banner printed to stdout contains the token, the local URL, and a prompt to fetch `/llms-local.txt`. An AI running `./run.sh` in a session reads that output, has the token, and can immediately pull its operating instructions — no QR code, no browser needed.

`/llms.txt` is a plain-text description of the available MCP tools and API, intended for AI assistants. `/llms-local.txt` (server-only) includes extended development instructions. Both are generated at request time from templates in `nicetransfer.py`.

In practice, AI access currently works from the server device — a desktop AI like Claude Code can connect directly. Remote clients on the same Wi-Fi can reach the browser UI, but most hosted AI assistants can't reach a local network address. As local AI apps and on-device models become more common, the discovery layer is already in place.
